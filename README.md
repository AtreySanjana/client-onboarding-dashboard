# Client Onboarding Dashboard

A simple dashboard prototype designed to simulate how client onboarding and implementation progress can be tracked in a service-based business.

This isn’t just a UI exercise — it’s meant to reflect how onboarding actually works behind the scenes: multiple clients, different stages, varying levels of risk, and the need to quickly understand what’s going on.

---

## What this project shows

The dashboard focuses on a few key things that matter in real workflows:

- Tracking project progress across different clients  
- Understanding ROI and time saved from automation  
- Identifying at-risk clients early  
- Keeping visibility on tasks and onboarding stages  

The idea is to make it easy to answer:
> “What’s happening with this client right now, and does anything need attention?”

---

## Features

- Client switcher to simulate different onboarding scenarios  
- KPI cards (ROI, progress, time saved, status)  
- Task tracking with completion toggles  
- Stage-based filtering (completed / in-progress / pending)  
- Timeline view of onboarding phases  
- Notifications for overdue or at-risk clients  
- Interactive charts (ROI trend and task distribution)  
- Persistent sidebar layout (like a real internal tool)

---

## Simulated client scenarios

Instead of using generic data, the dashboard includes a few realistic situations:

**WanderNest Travels**  
A growing travel startup that improved efficiency by reducing manual itinerary work through structured workflows.

**SoloSoul Adventures**  
An onboarding flow with delays and drop-offs. Marked as “At Risk” to reflect real-world situations where things don’t go as planned.

**Luxury Trails Co.**  
A premium service where onboarding focuses more on personalization and client experience rather than speed.

---

## Tech stack

- HTML  
- CSS  
- JavaScript  
- Chart.js (for visualizations)

No frameworks used — kept intentionally simple to focus on structure and logic.

---

## Why this matters

Most dashboards show data.

This one is designed to show **context** — what the data actually means, where things are breaking, and what might need action.

That’s closer to how onboarding and operations roles work in practice.
